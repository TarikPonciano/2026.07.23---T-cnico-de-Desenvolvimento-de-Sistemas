Ponto de partida

Por que orientar a objetos?

Até agora você escreveu programas com variáveis soltas e funções separadas. Funciona — até o projeto crescer.

# Sem POO: dados de UM produto espalhados
nome_produto = "Camiseta"
preco_produto = 49.90
estoque_produto = 12

# e se forem 200 produtos? E um cliente? E um pedido?
# vira um mar de variáveis soltas...
def aplicar_desconto(preco, percentual):
    return preco - (preco * percentual / 100)
O problema

Código espalhado

Os dados de cada produto ficam soltos, e toda função precisa receber tudo como parâmetro. Difícil de manter e fácil de errar.

A ideia da POO

Juntar o que é parecido

Reunir os dados e as ações da mesma coisa em uma única estrutura organizada.

Em uma frase: POO é programar pensando em objetos do mundo real — um produto, um cliente, um pedido — em vez de pensar só em comandos isolados.
Conceito central

Classe é a fôrma. Objeto é o bolo.

A melhor forma de entender a diferença é com uma analogia.

📐 Classe A planta / o molde
Existe uma só
➜
🍰 Objetos Os produtos finais
Pode ter vários

A classe "Bolo" descreve o que todo bolo tem (sabor, peso) e o que ele faz (ser servido). Cada bolo que você assa a partir dela é um objeto diferente — com seu próprio sabor e peso.

Outras fôrmas do dia a dia: a planta de uma casa (classe) e as casas construídas (objetos); o formulário de cadastro (classe) e cada cadastro preenchido (objetos).
Definições

As definições formais

Agora que a intuição está formada, vamos nomear as coisas com precisão.

Classe

O modelo / molde

É um modelo que define um tipo de objeto. Ela diz quais dados (atributos) e quais ações (métodos) os objetos daquele tipo terão. A classe sozinha é só a descrição — como uma receita que ainda não virou bolo.

Objeto

A instância concreta

É uma instância criada a partir de uma classe. Existe de verdade na memória e tem valores próprios. Instanciar é justamente o ato de criar um objeto a partir da classe.

Vocabulário que vamos usar o tempo todo:
  • Atributo — uma característica/dado do objeto. Ex.: o nome e o preço de um produto.
  • Método — uma ação/comportamento do objeto. Ex.: aplicar desconto.
  • Instância — outro nome para "objeto criado a partir da classe".
Parte prática

Mão na massa

Vamos construir, passo a passo, a classe Produto de uma loja — e dar vida a ela criando objetos.

Passo 1 · A estrutura

Criando uma classe

class Produto:
    # construtor: roda quando o objeto é criado
    def __init__(self, nome, preco):
        self.nome = nome    # atributo
        self.preco = preco  # atributo

    # método: comportamento do produto
    def exibir(self):
        return f"{self.nome} — R$ {self.preco:.2f}"
  • class declara a classe. Por convenção, o nome começa com letra maiúscula (Produto).
  • __init__ é o construtor: roda sozinho no momento em que o objeto nasce.
  • self é o próprio objeto falando de si mesmo — toda informação dele mora em self.
  • Os atributos self.nome e self.preco guardam os dados de cada produto.
Passo 1 · Aprofundando

Afinal, o que é o self?

É a parte que mais confunde no começo — então vamos com calma.

  • Pense em self como a palavra "eu" dita pelo próprio objeto.
  • Quando escrevemos self.nome, é o objeto dizendo: "o meu nome".
  • Cada objeto tem o seu próprio self — por isso dois produtos podem ter nomes diferentes.
  • Ele é sempre o primeiro parâmetro dos métodos, mas o Python o preenche automaticamente quando você usa o ponto.
camiseta = Produto("Camiseta", 49.90)
caneca   = Produto("Caneca", 29.90)

# para a camiseta, self é a camiseta:
#   self.nome  -> "Camiseta"

# para a caneca, self é a caneca:
#   self.nome  -> "Caneca"

camiseta.exibir()  # self = camiseta
caneca.exibir()    # self = caneca
Regra prática: dentro da classe, tudo que pertence ao objeto leva self. na frente.
Passo 2 · Dando vida

Instanciando objetos

# criando (instanciando) objetos da classe
p1 = Produto("Camiseta", 49.90)
p2 = Produto("Caneca", 29.90)

# acessando atributos com o ponto
print(p1.nome)    # Camiseta
print(p2.preco)   # 29.9

# chamando um método
print(p1.exibir())
# Camiseta — R$ 49.90
  • Produto("Camiseta", 49.90) chama a classe como se fosse uma função — e isso cria o objeto.
  • Os argumentos vão direto para o __init__.
  • p1 e p2 são objetos independentes: cada um com seus próprios valores.
  • Usamos o ponto (.) para acessar atributos e chamar métodos.
Repare: uma única classe, dois objetos diferentes. Esse é o coração da POO.
Passo 3 · Comportamento

Métodos que fazem algo

Métodos não só mostram dados — eles podem alterar o objeto e tomar decisões.

class Produto:
    def __init__(self, nome, preco, estoque):
        self.nome = nome
        self.preco = preco
        self.estoque = estoque

    def aplicar_desconto(self, percentual):
        self.preco -= self.preco * percentual / 100

    def vender(self, qtd):
        if qtd <= self.estoque:
            self.estoque -= qtd
            return f"Vendido(s) {qtd} de {self.nome}."
        return "Estoque insuficiente!"
tenis = Produto("Tênis", 200.0, 5)

tenis.aplicar_desconto(10)
print(tenis.preco)   # 180.0

print(tenis.vender(3))
# Vendido(s) 3 de Tênis.
print(tenis.estoque)  # 2

print(tenis.vender(10))
# Estoque insuficiente!

Os métodos mudam o estado do objeto: o preço e o estoque do tênis se alteram a cada chamada.

Passo 4 · Outro contexto

O mesmo padrão, outro contexto

Toda classe segue a mesma receita. Veja uma ContaCorrente de banco.

class ContaCorrente:
    def __init__(self, titular, saldo=0):
        self.titular = titular
        self.saldo = saldo

    def depositar(self, valor):
        self.saldo += valor

    def sacar(self, valor):
        if valor <= self.saldo:
            self.saldo -= valor
        else:
            print("Saldo insuficiente.")
conta = ContaCorrente("Ana")

conta.depositar(500)
conta.sacar(200)
print(conta.saldo)   # 300

conta.sacar(1000)
# Saldo insuficiente.
  • Mesma estrutura: __init__ + atributos + métodos.
  • saldo=0 é um valor padrão: se não informar, começa em zero.
  • Trocou o contexto, mas o raciocínio é idêntico.
Boas práticas

Organizando o projeto

Por enquanto, sem pastas: separamos a classe em um arquivo e o programa principal em outro, na mesma pasta.

loja/ │ ├── classProduto.py # a classe Produto └── main.py # programa principal

Em classProduto.py:

class Produto:
    def __init__(self, nome, preco):
        self.nome = nome
        self.preco = preco

    def exibir(self):
        return f"{self.nome} — R$ {self.preco:.2f}"

Em main.py:

from classProduto import Produto

p1 = Produto("Camiseta", 49.90)
print(p1.exibir())
Por que separar? Manter a classe em classProduto.py e o código que a usa em main.py deixa tudo organizado. O main.py só precisa importar a classe com from classProduto import Produto.
Visão geral

A anatomia de uma classe

Tudo o que vimos, em um único mapa para consultar sempre.

class Produto:                  ← nome da classe (maiúscula)

    def __init__(self, nome, preco): ← construtor
        self.nome = nome             ← atributo
        self.preco = preco           ← atributo

    def exibir(self):              ← método
        return f"{self.nome}: {self.preco}"


p1 = Produto("Camiseta", 49.9)   ← objeto (instância)
p1.exibir()                       ← chamando o método
class → declara a classe __init__ → construtor self → o próprio objeto atributo → dado (self.x) método → ação (def) objeto → Produto(...)
